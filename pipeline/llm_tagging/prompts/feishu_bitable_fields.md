# 打标主路径：飞书多维表格 AI 字段（完全由飞书完成）

> **当前主路径（v2）**。打标全流程留在飞书里，不引入 Python 或外部 LLM API 参与推理——智谱 API
> 只是曾经考虑过的 Plan B，不作为与飞书并存的正式环节。飞书 AI 字段的吞吐/配额限制通过
> **分批次跑**解决（一次跑不完就分几批跑），不是切换成代码调用。

## 设计思路

只有 **1 个字段做真正的推理**（`profile_json`，一次性联合判断维度、情感、用户画像），其余字段是
从这份 JSON 结果里**抽取**关键值，不重新推理——避免"同一条评论被独立问 N 次，互相矛盾"的问题
（比如维度和情感如果分开问，容易前后对不上；抽取字段直接读第一次推理已经定好的结论，不会有
这个问题）。

## 建表前提

- 先把 `data/processed/reviews_clean.csv` 导入为一张多维表格，确认 `content` 列存在
- **先在小样本上跑通再上全量**：每个车型分层抽样 20~30 条，人工抽查 `profile_json` 里的
  `evidence` 是不是真的从原文摘录（不是编的），确认没问题后再分批对全量跑
- 字段建表顺序有先后依赖：先建好 `profile_json`，等它对所有目标行都生成完，再建后面的抽取字段
  （它们的提示词要引用 `profile_json` 的值，这一列没生成完之前抽取字段会读到空值）

## 1. `profile_json`（长文本，AI 字段，引用 `{{content}}`）

**维度闭集枚举**（已按实际跑出来的数据整理成闭集，DMR/SDA 回测要求同一维度前后统计口径一致，
不能中途改词表）：

```
续航表现 / 辅助驾驶 / 车机智能化 / 内饰质感 / 外观设计 / 空间表现 / 底盘操控 / 舒适性 /
动力性能 / 充电体验 / 配置丰富度 / 性价比 / 用车成本 / 售后权益 / 故障维修 / 品牌认知 / 其他
```

提示词：

```
你是一位专注于汽车行业的用户研究专家，擅长从车主评论中提取结构化的用户洞察。

分析下面这条车主口碑原文，严格按照 JSON Schema 输出，不要输出任何 JSON 以外的内容。

严格规则（违反将导致输出无效）：
1. 只能从原文中提取或合理推断信息，禁止编造任何未在原文中提及或暗示的内容
2. 原文完全没有任何相关线索（哪怕是间接线索）时，才填 null；只要原文出现相关的间接线索，
   就应该基于这些线索给出最合理的推断，不要因为线索不是直接明说就默认填 null
3. evidence 字段有严格要求：
   - 只有当你能从原文中找到一段真实存在的连续文字来支撑某个判断时，才填这段文字，
     必须是原文的逐字节选，不得改写、总结或换一种说法
   - 如果想不出这样一段真实存在于原文中的文字，该字段的 value 和 evidence 必须同时为 null，
     不允许只填 value 不填真实 evidence
   - 绝对不允许把本提示词模板里的说明文字本身（比如"原文引用"这几个字）当成 evidence 的
     内容抄进去——这是占位说明，不是可以输出的答案
   - evidence 必须和它所支撑的 value 逻辑上直接相关，不能用一段不相关的原文话硬凑
4. 情感强度（intensity）基于原文措辞力度判断，1=轻微，5=强烈
5. dimension 只能从下面给定的枚举里选，不得自造新词；一条评论可包含多个维度，每个维度单独
   输出一条记录；同一维度同时有正负面描述时，分别输出两条记录
6. 输出必须是合法的 JSON，不得包含任何注释或额外文字

【输出格式】
{
  "user_profile": {
    "life_stage": {"value": "已婚有娃 | 已婚无娃 | 未婚 | 退休 | null", "evidence": "原文中的真实片段，或 null"},
    "purchase_motivation": {"value": "家用 | 商务接待 | 个人通勤 | 换购升级 | 首次购车 | null", "evidence": "原文中的真实片段，或 null"},
    "purchase_stage": {"value": "已购车 | 意向中 | 对比选购 | null", "evidence": "原文中的真实片段，或 null"},
    "gender": {"value": "男 | 女 | null", "evidence": "原文中的真实片段，或 null"}
  },
  "dimension_sentiments": [
    {
      "dimension": "续航表现 | 辅助驾驶 | 车机智能化 | 内饰质感 | 外观设计 | 空间表现 | 底盘操控 | 舒适性 | 动力性能 | 充电体验 | 配置丰富度 | 性价比 | 用车成本 | 售后权益 | 故障维修 | 品牌认知 | 其他",
      "sentiment": "正面 | 负面 | 中性",
      "intensity": 1,
      "evidence": "原文中的真实片段（必填，不得为 null）"
    }
  ],
  "key_highlights": ["最多 3 条原文直接引用，无则 []"],
  "key_pain_points": ["最多 3 条原文直接引用，无则 []"],
  "usage_scenarios": ["通勤 | 家庭出行 | 长途自驾 | 商务接送 | 其他"],
  "meta": {"confidence": "high | medium | low", "confidence_reason": "简述可推断程度"}
}

【评论原文】
{{content}}
```

## 2~5. 单选抽取字段（引用 `{{profile_json}}`）

四个字段提示词结构一样，只改字段名，选项分别对应 schema 里的枚举 + "无法判断"：

```
下面是一段结构化 JSON。直接读出 user_profile.life_stage.value 的值并返回对应选项，
不要重新分析原文，也不要输出 JSON 本身。如果值是 null，返回"无法判断"。

【JSON】
{{profile_json}}
```

同样模式建另外三个：`purchase_motivation`（选项：家用/商务接待/个人通勤/换购升级/首次购车/无法判断）、`purchase_stage`（选项：已购车/意向中/对比选购/无法判断）、`gender`（选项：男/女/无法判断）。

## 6~8. 多选抽取字段：`positive_dimensions` / `negative_dimensions` / `neutral_dimensions`（引用 `{{profile_json}}`）

这三个字段是为了让 DMR/SDA 回测直接读列，不用再解析 JSON——和 `data/reviews tagged.csv`（v1
遗留数据）是同一套字段名，格式对上了下游脚本不用改。

```
下面是一段结构化 JSON。从 dimension_sentiments 数组里，找出 sentiment 值为"正面"的所有
条目，把它们的 dimension 值作为选项返回（可多选）。不要重新分析原文，也不要输出 JSON 本身。
如果没有符合条件的条目，不选任何选项。

【JSON】
{{profile_json}}
```

`negative_dimensions` 把"正面"换成"负面"，`neutral_dimensions` 换成"中性"，其余不变。

## 关于吞吐/配额限制

飞书 AI 字段规模化跑的时候会遇到吞吐/配额限制（实测 1375 条跑到 1286 条时配额耗尽）。解决方式
是**分批次跑**：配额恢复后继续跑剩余部分，不需要切换成代码调用 API。跑的时候留意剩余配额，
避免像第一次一样跑到一半断在中间、还漏跑了情感字段。

## 关于 age

不新增这个字段。车评文本里能间接推断年龄的线索覆盖率实测只有约 2.7%，分母太小做不出有统计
意义的下钻，硬做只会是模型瞎猜，污染人群地图的可信度。方案里把 age 标注为"当前 UGC 文本数据
源覆盖弱，需接入交强险/车联网行为数据补全"的路线图项，不在本轮实现范围内。

## 质量检查（跑完小样本后）

- `profile_json` 里各字段 null 占比是否过高（> 60%），过高说明 prompt 或数据质量有问题
- 抽 5~10 条人工核对 `profile_json` 里的 `evidence`/`key_highlights`/`key_pain_points` 是否真的是原文摘录，而不是模型改写或编造——这是 v1 方案（`tagger.py`）暴露过证据造假问题后必须坚持的检查项，飞书 AI 字段没有代码层面的自动校验，人工抽检是唯一的质量把关手段
- 抽几条对比抽取字段的值和 `profile_json` 里对应的 value 是否一致（应该 100% 一致，因为是直接读取不是重新判断；如果不一致，说明抽取字段的提示词没写对，模型在"重新分析"而不是"读取"）
