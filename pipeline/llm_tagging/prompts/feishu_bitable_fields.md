# 打标主路径：飞书多维表格 AI 字段 + Python 补全

> **当前主路径（v2）**。原先设想"整段 JSON 全部交给飞书 AI 字段推理"的方案，实测卡在飞书 AI
> 字段的吞吐/配额限制上（1375 条评论跑到第 1286 条时配额耗尽，情感字段甚至完全没跑出结果）。
> 现在的分工是：**飞书只做维度提取**（它确实做得比固定枚举好），**情感方向和用户画像交给
> Python 批量调 LLM API 补**（不受飞书 AI 字段吞吐限制，且加了 evidence 原文子串强校验）。
> 两部分结果最终合并写回同一张飞书多维表格，产品经理只看这张表，不接触内部分工。

---

## Part A：飞书多维表格 AI 字段——「评论打标」（维度提取）

把 `data/processed/reviews_clean.csv` 导入多维表格后，在表里新增一个长文本 AI 字段，引用
`{{评论内容}}`。

**维度枚举**（已按目前实际跑出来的 1286 条数据里出现过的词汇整理成闭集，后续无论是飞书继续
跑剩余的 89 条、还是 Python 补全脚本从头打标未覆盖记录，都要用同一套词表，否则同一个维度会
因为用词不一致被统计成两类，回测的 DMR/SDA 会失真）：

```
续航表现 / 辅助驾驶 / 车机智能化 / 内饰质感 / 外观设计 / 空间表现 / 底盘操控 / 舒适性 /
动力性能 / 充电体验 / 配置丰富度 / 性价比 / 用车成本 / 售后权益 / 故障维修 / 品牌认知 / 其他
```

提示词：

```
你是一位专注于汽车行业的用户研究专家。阅读下面这条车主口碑原文，判断它涉及了哪些方面，
只能从下面给定的枚举里选，不得自造新词，一条评论可以涉及多个方面：

续航表现 / 辅助驾驶 / 车机智能化 / 内饰质感 / 外观设计 / 空间表现 / 底盘操控 / 舒适性 /
动力性能 / 充电体验 / 配置丰富度 / 性价比 / 用车成本 / 售后权益 / 故障维修 / 品牌认知 / 其他

只输出命中的方面名称，用逗号分隔，不要输出任何解释或额外文字。

【评论原文】
{{评论内容}}
```

**建表提示**：先在小样本（每车型 20~30 条）上跑通、抽查结果没问题再上全量；飞书 AI 字段有
吞吐配额限制，全量建议分批跑，跑之前留意剩余配额，避免像这次一样跑到一半断在中间。

## Part B：Python 补全脚本——情感方向 + 用户画像

这部分不在飞书里配置，是一个独立的 Python 脚本（替代已废弃的 `tagger.py`），输入是飞书导出
的数据（`评论ID` / `评论内容` / `评论打标`），对每条记录调用 LLM API，只问两件事：

1. **飞书已经标出的这几个方面，各自是正面/负面/中性**（不重新判断"涉及哪些方面"，飞书已经
   做完这步，这样调用更快更准，因为方面是给定的而不是要模型自己猜）
2. **用户画像**：life_stage / purchase_motivation / purchase_stage / gender

对飞书还没覆盖到的记录（配额耗尽前没跑到的那部分），脚本要连维度提取一起从头做。

### Prompt 设计

```
你是一位专注于汽车行业的用户研究专家，擅长从车主评论中提取结构化的用户洞察。

分析下面这条车主口碑原文，严格按照 JSON Schema 输出，不要输出任何 JSON 以外的内容。

严格规则（违反将导致输出无效）：
1. 只能从原文中提取或合理推断信息，禁止编造任何未在原文中提及或暗示的内容
2. 原文完全没有任何相关线索（哪怕是间接线索）时，才填 null；只要原文出现相关的间接线索，
   就应该基于这些线索给出最合理的推断，不要因为线索不是直接明说就默认填 null
3. evidence 字段有严格要求：
   - 只有当你能从原文中找到一段真实存在的连续文字来支撑某个判断时，才填这段文字，必须是
     原文的逐字节选，不得改写、总结或换一种说法
   - 如果想不出这样一段真实存在于原文中的文字，该字段的 value 和 evidence 必须同时为 null
   - 绝对不允许把本提示词模板里的说明文字本身（比如"原文引用"这几个字）当成 evidence 的
     内容抄进去——这是占位说明，不是可以输出的答案
   - evidence 必须和它所支撑的 value 逻辑上直接相关，不能用一段不相关的原文话硬凑
4. 情感强度（intensity）基于原文措辞力度判断，1=轻微，5=强烈
5. 输出必须是合法的 JSON，不得包含任何注释或额外文字

【已知信息：这条评论已经识别出涉及以下方面】
{{评论打标}}

【输出格式】
{
  "user_profile": {
    "life_stage": {"value": "已婚有娃 | 已婚无娃 | 未婚 | 退休 | null", "evidence": "原文中的真实片段，或 null"},
    "purchase_motivation": {"value": "家用 | 商务接待 | 个人通勤 | 换购升级 | 首次购车 | null", "evidence": "原文中的真实片段，或 null"},
    "purchase_stage": {"value": "已购车 | 意向中 | 对比选购 | null", "evidence": "原文中的真实片段，或 null"},
    "gender": {"value": "男 | 女 | null", "evidence": "原文中的真实片段，或 null"}
  },
  "dimension_sentiments": [
    {
      "dimension": "上面【已知信息】列出的方面之一，逐一都要输出",
      "sentiment": "正面 | 负面 | 中性",
      "intensity": 1,
      "evidence": "原文中的真实片段（必填，不得为 null）"
    }
  ]
}

【评论原文】
{{评论内容}}
```

如果这条记录不在飞书已覆盖的 1286 条里（即【已知信息】为空），改用完整版 Prompt——在上面
基础上去掉"【已知信息】"这段，把 `dimension_sentiments` 的 `dimension` 枚举换成 Part A 里
那 17 类的完整列表，让模型自己判断涉及哪些方面（等价于把 Part A 和这一步合并成一次调用）。

### 代码侧必须实现的校验（对应 v1 暴露的问题）

- `evidence` 非 null 时，必须是对应记录 `content` 字段的真实子串，不满足则把该字段整体
  改判为 `null`（不是丢弃整条记录，是这一个字段作废），并计入"需人工复核"名单
- 抽 5~10% 结果人工核对 evidence 是否真的是原文摘录

### 输出

合并飞书的「评论打标」列 + 本脚本补出的情感/画像结果，写成 `data/processed/reviews_tagged.jsonl`
（`backtest/compute_metrics.py` 直接认这个格式），同时把结果通过飞书开放平台 API 写回原表。

## 关于 age

不新增这个字段。车评文本里能间接推断年龄的线索覆盖率实测只有约 2.7%，分母太小做不出有统计
意义的下钻，硬做只会是模型瞎猜，污染人群地图的可信度。方案里把 age 标注为"当前 UGC 文本数据
源覆盖弱，需接入交强险/车联网行为数据补全"的路线图项，不在本轮实现范围内。
